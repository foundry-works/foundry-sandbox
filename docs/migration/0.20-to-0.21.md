# Migrating from 0.20.x to 0.21.x

> **Deprecation notice (2026-04-22):** The 0.20.x migration path — including `cast migrate-to-sbx`, `cast migrate-from-sbx`, and rollback support — will be removed in **0.23.0** (target: October 2026). If you are still running 0.20.x, migrate before that release. After removal, 0.20.x metadata and presets will no longer be importable.

This guide covers upgrading from `foundry-sandbox` 0.20.x (docker-compose backend) to 0.21.x (Docker `sbx` microVM backend).

## Overview

Version 0.21.x replaces the entire docker-compose backend with Docker's native `sbx` microVM backend, removing ~67k lines of proxy infrastructure. Your sandboxes, presets, and credentials are automatically migrated.

**Key changes:**
- MicroVM isolation (hypervisor-level) replaces Docker containers
- `sbx` handles network policy, secret storage, and credential injection
- Git safety is now a standalone service (`foundry-git-safety`)
- No more unified proxy, mitmproxy, Squid, or docker-compose generation

## Prerequisites

- Docker Desktop 4.37+ with the Sandboxes feature enabled, or standalone `sbx` v0.26.1+
- `foundry-sandbox` 0.21.x installed (`pip install --upgrade foundry-sandbox`)
- No running sandboxes (destroy or stop all before migrating)

## Migration Steps

### 1. Destroy running sandboxes

```bash
cast destroy-all
```

### 2. Run the migration

```bash
cast migrate-to-sbx
```

This command:
1. Scans `~/.sandboxes/` for existing sandbox metadata and presets
2. Creates a timestamped snapshot in `~/.sandboxes/.migration-snapshots/`
3. Pushes API keys from your environment to `sbx` secrets
4. Converts sandbox metadata from `metadata.env`/old `metadata.json` to `SbxSandboxMetadata`
5. Converts presets (maps old fields, drops removed flags)

### 3. Review the output

Check for warnings about:
- **Missing credentials** — set the env var and run `cast refresh-credentials`
- **Dropped fields** — features like `--mount`, `--network`, `--with-ssh` are no longer available

### 4. Verify

```bash
cast list
cast config
```

### 5. Start migrated sandboxes

Each migrated sandbox needs to be fully provisioned before use. This happens automatically on first start:

```bash
cast start my-sandbox
```

The first start will:
1. Fetch or reuse the existing bare repo
2. Create or reuse the existing worktree
3. Create the sbx microVM sandbox
4. Provision git safety (HMAC secrets, wrapper injection, registration)

This may take 30-60 seconds per sandbox. Subsequent starts are instant.

Existing bare repos and worktrees from 0.20.x are reused — no re-clone needed.

## Dry Run

To preview changes without making them:

```bash
cast migrate-to-sbx --plan
```

## Breaking Changes

### Removed Commands

| Command | Replacement |
|---------|-------------|
| `cast build` | Managed by `sbx` templates |
| `cast prune` | Handled by `sbx` |
| `cast upgrade` | `sbx` handles updates |

### Removed Flags for `cast new`

| Flag | Notes |
|------|-------|
| `--mount`, `-v` | `sbx` uses file sync instead of bind mounts |
| `--network`, `-n` | Replaced by `sbx policy` commands |
| `--with-ssh` | `sbx` handles SSH differently |
| `--no-isolate-credentials` | `sbx` always isolates credentials |
| `--sparse` | Deferred to future implementation |
| `--pre-foundry` | Docker-compose specific |
| `--compose-extra` | Docker-compose specific |
| `--allow-dangerous-mount` | No bind mounts in `sbx` |
| `--anthropic-base-url` | `sbx` handles credential routing |

### New Flags for `cast new`

| Flag | Description |
|------|-------------|
| `--agent` | Select agent type: claude, codex, copilot, gemini, kiro, opencode, shell |
| `--name` | Override auto-generated sandbox name |
| `--save-as` | Save configuration as named preset |
| `--skip-key-check` | Skip API key validation |
| `--with-opencode` | Enable OpenCode setup |
| `--with-zai` | Enable ZAI Claude alias |

### Credential Changes

| 0.20.x | 0.21.x |
|--------|--------|
| Unified proxy (mitmproxy) injects credentials | `sbx secret set -g` stores credentials |
| Tavily, Perplexity, Semantic Scholar auto-injected | Reverse proxy via `config/user-services.yaml` (auto-migrated) |
| `--no-isolate-credentials` bypasses injection | Always isolated, no bypass |

### Network Changes

| 0.20.x | 0.21.x |
|--------|--------|
| `--network limited` (default) | `--network balanced` (default) |
| `--network host-only` | `--network allow-all` |
| `--network none` | `--network deny-all` |

## Rollback

To revert to 0.20.x:

```bash
cast migrate-from-sbx
pip install foundry-sandbox==0.20.15
```

The `migrate-from-sbx` command restores your metadata from the snapshot created during migration. You can also specify a specific snapshot:

```bash
cast migrate-from-sbx --snapshot-dir ~/.sandboxes/.migration-snapshots/20260420T120000Z
```

## FAQ

**Q: What happened to my user-defined service credentials (Tavily, Perplexity)?**
A: User-defined credential injection is supported via a reverse proxy on the foundry-git-safety server. Declare services in `config/user-services.yaml` and they are migrated automatically by `cast migrate-to-sbx`.

**Q: Do I need to re-clone my repos?**
A: No. Bare repos and worktrees in `~/.sandboxes/` are preserved and reused when you run `cast start` on a migrated sandbox. The migration only converts metadata files; the actual sbx sandbox is created on first start.

**Q: What if the migration is interrupted?**
A: Re-run `cast migrate-to-sbx`. It detects the migration lock and offers to continue. If you prefer to start over, run `cast migrate-from-sbx` to rollback first.

**Q: Can I run both 0.20.x and 0.21.x sandboxes simultaneously?**
A: No. See ADR-009 for the rationale. The snapshot/rollback mechanism provides safety without dual-mode complexity.
